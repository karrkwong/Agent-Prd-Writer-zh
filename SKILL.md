# Agent PRD Writer zh

当任务是创建或改进 Agent 产品需求文档时，使用本 skill。

本 skill 面向 Agent 产品，不面向普通功能需求。出现以下情况时应使用：

- 产品包含一个或多个 Agent，这些 Agent 会观察上下文、生成结果、调用工具或执行动作。
- 产品是一组按业务场景或职责拆分的 Agent 套件。
- 产品涉及跨系统流程，例如邮箱、日历、文档、文件、任务、CRM、工单、审批、聊天或内部系统。
- 产品需要定义高影响动作的用户确认规则。
- 产品涉及 Agent 交接、调度、记忆、权限、工具调用、评估、安全、审计或失败恢复。

## 使用方式

1. 先读取用户提供的业务背景、已有文档、demo、截图、设计文件或代码。
2. 判断产品应是单一 Agent，还是 Agent 套件。
3. 如果信息不足，先提出关键问题。不要在范围不清时直接写完整 PRD。
4. 如果用户要求完整输出，按 `references/agent-prd-template.md` 生成 PRD。
5. 如果用户要求开发可开工，必须同时使用 `references/engineering-contracts.md`，补齐运行契约、工具契约、交接 payload、状态转移、日志审计和测试计划。
6. 如果用户要求参考样例，再读取 `references/examples.md`。该文件只放短样例。
7. 输出前按质量检查清单自检。

## 核心判断

先判断产品应该是单一 Agent，还是 Agent 套件。

如果业务场景横跨多个任务、数据源、工具、权限或风险等级，优先定义为 Agent 套件。

如果是 Agent 套件，需要定义：

- 面向用户的总入口 Agent 或调度入口。
- 多个职责清晰的场景 Agent。
- 涉及外部动作时的工具执行 Agent 或执行层。
- Agent 之间的交接规则。
- 权限、确认、失败恢复和评估规则。

不要把互不相同的职责压进一个泛化 Agent。能拆清楚的职责应拆开，便于定义、开发、测试和治理。

## 写作前检查

如果以下信息不清楚，应先提问，再写完整 PRD：

- 目标用户。
- 业务场景。
- 产品范围。
- 输出是全量规划，还是首版落地。
- 真实数据源和工具接入范围。
- 哪些动作是真实执行，哪些动作只是 mock 或 demo。
- 用户确认要求。
- 上线或演示场景。
- 评估口径和验收标准。

如果只有部分信息缺失，可以继续写 PRD，但必须把未确认项标记为 `待确认`。不要编造事实、数据指标、系统接入能力或用户反馈。

## 写作风格

默认使用清晰、直接、专业的中文。

要求：

- 写成内部产品文档，不写成说明文或宣传文。
- 产品判断要明确。
- 需求、契约和验收标准尽量可测试。
- 结构复杂的内容优先用表格。
- 使用具体事实、假设和决策。
- 不确定内容要明确标记。
- 避免口号化表达和宽泛判断。
- 避免使用 `简单来说`、`你可以这样理解`、`本质上`、`可以看到`、`先说结论`、`根据你的反馈`。
- 避免使用 `不是 A，而是 B`、`不取决于 A，而取决于 B`、`真实价值点不在 A，而在 B` 这类固定句式。
- 避免使用 `赋能`、`抓手`、`闭环`、`底层逻辑`、`控制点`、`边界`、`关键入口`、`能力分层` 等抽象词，除非上下文明确解释其含义。

## 需要读取的 references

- 写完整 PRD：读取 `references/agent-prd-template.md`。
- 写开发可开工的 PRD：同时读取 `references/engineering-contracts.md`。
- 用户要求参考样例时：再读取 `references/examples.md`。

## 质量检查

交付前检查：

- 是否明确单一 Agent / Agent 套件。
- 是否包含非目标范围。
- 是否有每个 Agent 的触发、输入、输出、工具、异常、验收。
- 是否有 Agent 运行契约。
- 是否有工具调用契约。
- 是否有交接 payload。
- 是否有状态机和状态转移规则。
- 是否有权限、确认、风险动作分级。
- 是否有记忆与数据保留规则。
- 是否有日志与审计要求。
- 是否有评估数据集和通过阈值。
- 是否有工程测试计划。
- 是否标记所有待确认项。
- 是否避免编造数据、系统能力或用户反馈。
