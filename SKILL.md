# Agent PRD Writer zh

当任务是创建或改进 Agent 产品需求文档时，使用本 skill。

适用场景：

- 产品包含一个或多个 Agent，会观察上下文、生成结果、调用工具或执行动作。
- 产品涉及跨系统流程，例如邮箱、日历、文档、任务、CRM、审批、聊天或内部系统。
- 产品需要定义用户确认、人工介入、权限控制、高影响动作、评估或失败恢复。

## 输出分层

默认输出 `PM 可负责的 Agent PRD`。只有用户明确要求“开发可开工”“工程可落地”“可以直接开发”时，才追加工程契约和开发交付包。

| 输出层 | 何时输出 | 内容边界 |
|---|---|---|
| PM PRD | 默认 | 产品目标、用户场景、Agent 角色、关键流程、权限确认、验收、风险、版本计划 |
| 工程契约 | 用户要求可开发时 | schema、工具契约、handoff、状态转移、日志审计、测试计划 |
| 开发交付包 | 用户要求交付工程任务时 | 技术栈、模块拆分、mock、运行方式、部署、监控、CI、回滚 |

## 使用方式

1. 读取用户提供的背景、已有文档、demo、截图、设计文件或代码。
2. 判断产品是单一 Agent 还是 Agent 套件。
3. 判断当前输出是方向讨论、PM PRD，还是开发可开工文档。
4. 信息不足时先问关键问题；可以继续写时，必须标记 `待确认`。
5. 写完整 PRD 时读取 `references/agent-prd-template.md` 和 `references/product-patterns.md`。
6. 写开发可开工文档时，再读取 `references/engineering-contracts.md` 和 `references/development-handoff.md`。
7. 输出前按质量检查和人写感检查自检。

## 核心判断

先判断产品形态：

- 单一 Agent：任务边界清楚，输入、工具、风险等级相对集中。
- Agent 套件：场景横跨多个任务、数据源、工具、权限或风险等级。

如果是 Agent 套件，需要写清用户看到的总入口、职责清楚的场景 Agent、必要原子能力、外部动作执行层、Agent 交接方式、权限确认和失败恢复规则。

不要把互不相同的职责压进一个泛化 Agent。能拆清楚的职责应拆开，便于定义、开发、测试和治理。

## 写作前检查

以下信息会直接影响 PRD 质量：目标用户、业务场景、首版范围、Agent 用户可见角色、输入来源、最终输出、真实数据源和工具接入范围、mock 范围、用户确认和人工介入要求、上线或演示场景、验收口径。

缺失信息可以继续写，但必须标记为 `待确认`。不要编造事实、数据指标、系统接入能力或用户反馈。

## 写作风格

默认使用清晰、直接、专业的中文。写成内部产品文档，不写成说明文、宣传文或模板填空。

要求：

- 产品判断明确，需求和验收标准尽量可测试。
- 使用具体事实、假设和决策。
- 保留业务现场语言，不把所有内容改写成抽象 Agent 术语。
- 表格只用于比较、拆解、验收和责任分工；能用短段落讲清的内容不要强行表格化。
- 避免口号化表达、宽泛判断和过度安全正确的重复句。
- 避免 `简单来说`、`你可以这样理解`、`本质上`、`可以看到`、`先说结论`、`根据你的反馈`。
- 避免 `不是 A，而是 B`、`不取决于 A，而取决于 B`、`真实价值点不在 A，而在 B` 这类固定句式。
- 避免 `赋能`、`抓手`、`闭环`、`底层逻辑`、`控制点`、`边界`、`关键入口`、`能力分层` 等抽象词，除非上下文明确解释其含义。

## 人写感输出规则

本 skill 默认输出像产品负责人写给项目团队的工作文档。文档先推动当前项目判断，再按需要补工程细节。完整模板不是目标。

写作时遵守以下规则：

- 不要为了显得完整，把所有模板章节平均写满。
- 主文档优先讲清产品主线、用户场景、当前取舍、首版范围和关键风险。
- 工程契约、schema、handoff、日志、测试矩阵默认放在工程附录。
- 首版主线、真实用户输入、关键流程、待确认依赖、高影响动作写深；远期能力、通用安全、通用日志、非首版 Agent 写短或放附录。
- 不确定内容写成当前项目状态和临时处理方式，例如“真实邮箱接入待工程确认，Demo 先用 mock 邮件跑通流程”。
- 同一类安全、权限、日志规则不要在多个章节重复。主文档讲原则，附录写细则。

## 质量检查

交付前检查：

- 是否明确单一 Agent / Agent 套件。
- 是否说明用户可见角色、首版主线和非目标范围。
- 是否有关键 Agent 的触发、输入、输出、工具、异常、验收。
- 是否有必要的权限、确认、HITL、降级和验收规则。
- 是否标记所有待确认项，没有编造数据、系统能力或用户反馈。
- 是否把工程细节放到附录，没有压住正文。
- 是否有真实业务词、用户输入、系统触发或项目协作语境。
- 是否读起来像项目负责人推进事情，而不是模板自动补全。
