# Agent PRD 模板

本模板用于输出完整 Agent PRD，但不要求所有章节等量展开。先判断用户需要的是方向讨论稿、PM PRD，还是开发可开工文档。

## 使用模板前的判断

- 方向讨论稿：只讲产品判断、主线场景、首版范围、风险和待确认问题。
- PM PRD：补齐 Agent 角色、用户场景、关键流程、权限确认、验收和版本计划。
- 开发可开工 PRD：在 PM PRD 后追加工程附录，展开 schema、工具契约、handoff、状态机、日志和测试计划。

正文要像项目文档，附录才像工程合同。

# 1. 产品判断与定位

说明业务场景、当前用户流程、主要问题，以及为什么适合引入 Agent。

写清产品是单一 Agent 还是 Agent 套件；用户看到什么入口；系统内部大致如何分工。不要一开始就铺完整工程架构。

# 2. 用户与核心场景

写目标用户、典型任务、当前替代方案和核心场景。每个核心场景至少给 1 个真实用户输入、业务触发或系统事件样例。

# 3. 首版主线流程

按用户实际使用顺序写：触发来源、Agent 先处理什么、用户看到什么、哪些信息需要补齐、哪些动作需要确认、失败后如何继续。

# 4. Agent 分工

如果是 Agent 套件，写清：

- 总入口 Agent：负责识别任务、展示状态、调度子 Agent、管理确认。
- 场景 Agent：负责具体任务，不包揽全流程。
- 工具执行层：只处理用户确认后的外部动作。

表格只列关键 Agent，不必把远期 Agent 全部写满。

# 5. 关键需求

每个关键 Agent 单独成节。默认写以下内容即可：

| 模块 | 需求 |
|---|---|
| 用户可见角色 |  |
| 触发条件 |  |
| 输入 |  |
| 输出 |  |
| 关键能力 |  |
| 工具 / 数据源 | 真实接入 / mock / 待确认 |
| 用户确认 |  |
| 异常与降级 |  |
| 验收标准 |  |

Prompt 契约、运行契约、详细 schema 放到工程附录，不默认压进主文档。

# 6. 权限、确认与人工介入

写清哪些动作可以自动生成，哪些动作必须用户确认，哪些情况需要人工介入。高影响动作、来源不足、权限不足、用户主动转人工、系统无法判断都要覆盖。

# 7. 验收与指标

写能判断通过与否的验收标准。Demo 可以用可控样本和人工验收；MVP 以后再写量化阈值和评估集。

# 8. 风险、依赖与开放问题

风险写真实会影响项目推进的内容，不要默认铺满通用风险。未确认内容写成当前项目状态和临时处理方式。

# 9. 版本计划与非目标范围

写 Demo / MVP / v1 的主要范围。非目标范围重点覆盖：首版不做泛聊天、自动高影响执行、未授权读取、默认长期记忆、敏感信息外层展示。不要把非目标范围写成通用安全条款合集。

# 10. 结论与建议

短段落收束：推荐产品形态、首版范围、最重要的待确认决策，以及是否已经达到开发可开工状态。

# 工程附录（开发可开工时再展开）

开发可开工文档再追加以下内容，细节参考 `references/engineering-contracts.md`：

- Agent 运行契约。
- 输入 / 输出 schema。
- 工具调用契约。
- Agent 交接 payload。
- 状态转移规则。
- 记忆与数据策略。
- 日志、审计与可观测性。
- 工程测试计划。
