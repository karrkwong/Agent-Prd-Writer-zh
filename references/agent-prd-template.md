本模板用于输出完整 Agent PRD。不要把它简化成普通功能 PRD。Agent PRD 必须写清 Agent 职责、工具、权限、确认、状态、失败、评估和工程可验收规则。

# 1. 产品判断与定位

## 1.1 产品背景

说明业务场景、当前用户流程、主要问题，以及为什么适合引入 Agent。

## 1.2 产品定位

说明产品是什么，是单一 Agent 还是 Agent 套件，面向用户的入口是什么，内部 Agent 如何分工，用户看到什么，系统内部如何运行。

|---|---|---|

| 场景 Agent 层 |  |  |
| 工具与数据层 |  |  |
| 权限与评估层 |  |  |

## 1.3 产品原则

写清：

- 哪些内容可以自动生成，哪些动作必须用户确认。
- 来源和不确定性如何标注。
- 隐私和敏感信息如何展示。
- Agent 职责如何拆分。
- 首版范围如何收敛。

# 2. 目标与成功指标

## 2.1 产品目标

| 目标 | 说明 | 验证方式 |
|---|---|---|

目标必须可验证，例如降低准备时间、提升信息完整度、减少遗漏、防止未授权执行、提升用户信任。

## 2.2 成功指标

| 指标 | 定义 | 建议口径 |
|---|---|---|

根据场景选择 Agent 专属指标：

- 任务识别准确率。
- 信息提取准确率。
- 来源覆盖率。
- 状态可解释率。
- 工具调用成功率。
- 行动项识别准确率。
- 高影响动作误执行次数。
- 失败可恢复率。
- 建议采纳率。

## 4.3 Agent 运行契约总览

本节引用 `references/engineering-contracts.md` 的 Agent 运行契约。每个 Agent 至少要补齐：

| 字段 | 说明 |
|---|---|
| 触发事件 | 由什么事件触发，例如用户点击、定时提醒、数据进入、上游 Agent 交接。 |
| 输入 schema | 输入字段、字段类型、必填项、允许为空项。 |
| 输出 schema | 输出字段、字段类型、必填项、置信度或来源字段。 |
| 可调用工具 | 允许调用的工具列表。 |
| 工具调用上限 | 单次任务最多调用次数，避免无限循环。 |
| 超时策略 | 单个工具和整体任务的超时时间。 |
| 重试策略 | 哪些错误可重试，最多重试几次。 |
| 停止条件 | 成功、失败、用户取消、信息不足、超时等停止条件。 |
| 需要用户确认的条件 | 哪些输出或动作必须用户确认。 |
| 异常降级方式 | 工具失败、权限不足、来源不足时如何降级。 |

# 5. Agent 分工与交接规则

## 5.1 交接原则

写清 Agent 之间交接什么、交接内容必须包含哪些结构化字段、哪些情况下停止交接、来源不足如何标记、高影响动作由哪个 Agent 或哪一层处理。

## 5.2 交接路径

| 触发条件 | 当前 Agent | 交给谁 | 交接内容 |
|---|---|---|---|

## 5.3 Agent 交接 payload schema

交接 payload 必须结构化。参考：

```json
{
  "task_id": "",
  "user_intent": "",
  "source_context": [],
  "current_agent": "",
  "next_agent": "",
  "handoff_reason": "",
  "required_tools": [],
  "risk_level": "",
  "needs_user_confirmation": true,
  "missing_info": [],
  "expected_output": ""
}
```

字段规则：

| 字段 | 必填 | 允许为空 | 说明 |
|---|---|---|---|
| task_id | 是 | 否 | 当前任务唯一标识。 |
| user_intent | 是 | 否 | 用户目标或系统识别出的任务意图。 |
| source_context | 是 | 可为空数组 | 来源上下文列表。来源不足时填空数组，并在 missing_info 标记缺口。 |
| current_agent | 是 | 否 | 当前 Agent 名称。 |
| next_agent | 是 | 否 | 接收方 Agent 名称。 |
| handoff_reason | 是 | 否 | 交接原因。 |
| required_tools | 是 | 可为空数组 | 下游可能需要的工具。 |
| risk_level | 是 | 否 | low / medium / high。 |
| needs_user_confirmation | 是 | 否 | 是否需要用户确认。 |
| missing_info | 是 | 可为空数组 | 缺失信息。来源不足、权限不足、用户未授权都写在这里。 |
| expected_output | 是 | 否 | 下游应输出什么。 |

# 7. 自主等级与确认规则

## 7.1 动作分级

| 动作等级 | 示例 | 默认策略 |
|---|---|---|
| L0 展示 | 摘要、状态、来源 | 可自动生成 |
| L1 草稿 | 文档草稿、邮件草稿、待办草稿 | 可自动生成，但必须标记草稿 |
| L2 本地状态更新 | 任务卡片内状态变化 | 可在任务内自动更新 |
| L3 外部写入 | 创建日程、写入待办、保存文档 | 需要用户确认 |
| L4 对外动作 | 发送邮件、邀请外部联系人、共享文档 | 必须展示预览并确认 |
| L5 禁止动作 | 未经确认发送、绕过权限读取、自动对外承诺 | 禁止 |

## 7.2 确认规则

写清哪些动作需要确认、确认前展示哪些字段、用户修改草稿后是否需要再次确认、用户取消后如何处理、执行失败后如何恢复。

# 14. 非目标范围

Agent PRD 必须包含本章节。

本章节要明确首版不做什么、延后什么、禁止什么。尤其要覆盖：

- 首版不做泛聊天入口，Agent 必须围绕明确任务运行。
- 首版不允许自动对外执行高影响动作。
- 首版不允许未经授权读取邮箱、日历、文档、文件、通讯录或业务系统。
- 首版不允许未经确认跨系统写入。
- 首版不做默认长期记忆。确需记忆时，必须有用户授权、查看、修改和删除入口。
- 首版不把来源不足的信息写成确定结论。
- 首版不在锁屏、桌面、通知等外层入口展示敏感正文。
- 首版不做企业级管理员策略平台，除非当前项目目标明确要求。

# 15. 工程测试计划

开发前必须定义测试类型：

| 测试类型 | 测试目标 | 样本 / 方法 | 通过标准 |
|---|---|---|---|
| 单 Agent 单元测试 | 验证单个 Agent 的输入输出和异常处理。 | 结构化样本。 | 核心路径通过。 |
| 多 Agent 交接测试 | 验证 handoff payload 和状态继承。 | 多轮任务样本。 | 字段完整，交接正确。 |
| 工具调用测试 | 验证工具参数、返回值、超时和重试。 | mock + 真实工具。 | 调用符合契约。 |
| 权限与确认测试 | 验证高影响动作不能绕过用户确认。 | 权限矩阵。 | 误执行次数为 0。 |
| 失败恢复测试 | 验证权限不足、工具失败、来源不足时可降级。 | 失败注入。 | 用户可继续处理。 |
| 安全与隐私测试 | 验证敏感信息不外显、不被错误保存。 | 敏感样本。 | 不违反展示和存储规则。 |
| 回归测试 | 验证新版本不破坏已通过样本。 | 固定回归集。 | 通过率达到版本阈值。 |
| 端到端验收测试 | 验证完整用户主线。 | 真实或近真实任务。 | 主流程可完成。 |
