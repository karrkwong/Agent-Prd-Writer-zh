# Agent 产品表达参考

本文件只补强 Agent PRD 的产品表达，不重复完整 PRD 模板，也不替代工程契约。

## 1. 真实业务表达

Agent PRD 需要用具体场景支撑判断。优先写用户会说什么、系统先看到什么、Agent 先处理什么、哪些信息需要补齐、哪些动作需要确认。

推荐写法：

- 用户收到一封包含会议问题的邮件后，OfficeAgent 先提取问题、检查现有材料、标出缺口。
- 用户输入“预算 5 万，两个人，想出国看演唱会”后，Agent 先补齐时间、城市和艺人偏好。
- 系统识别到航班后，先生成出行卡片和出发前清单，不直接修改闹钟或叫车。

避免只写：

- Agent 基于多维上下文进行智能推荐。
- 系统通过能力编排完成任务闭环。
- 工具层提供全链路执行能力。

## 2. 保留业务词

优先保留资料中的真实词：banner、推荐问题、客服微信、服务部、一期纯文字、二期卡片、mock 邮件、静态路线、用户 query、demo 操作、团队依赖。

不要把这些词全部替换成“能力、链路、框架、治理、调度”等抽象词。

## 3. Agent 角色写法

Agent 角色说明用户把它当作什么工作对象使用。写清六件事即可：

- 角色名称：用户能理解的名称，例如会议助理、销售线索助理、出行管家。
- 替代或增强的人工任务：它接管或辅助哪类具体工作。
- 典型输入：用户给它什么材料、指令或上下文。
- 典型输出：它交付什么结果。
- 不能做的事：它不得承诺、不得执行或不得判断的范围。
- 用户信任依据：来源、预览、确认、可撤销、人工介入等机制。

## 4. 能力、Node、Prompt 的轻量写法

需要写清，但不要在主文档里铺满工程表格。

- 原子能力：写能力名称、职责、输入、输出、失败处理和归属 Agent。不要把完整流程写成一个能力。
- Node / 工作步骤：写进入条件、调用能力、产出、失败策略、是否需要 HITL。不绑定 LangGraph、ADK 或其他实现。
- Prompt 契约：写角色、任务目标、输入变量、输出格式、禁止事项、失败回复和 1-2 个短样例。
- HITL：写触发条件、用户可见入口、处理方式和结果回写。高风险、权限不足、系统无法判断、用户主动转人工都要覆盖。
- 埋点：只写与当前产品目标相关的事件，不要默认铺满通用埋点表。
- 用户可见降级：说明用户看到什么、还能做什么、系统保留什么、如何恢复。

## 5. AI 感写法对照

AI 感偏重：

> 系统通过用户入口层、场景 Agent 层、工具与数据层和权限评估层，实现办公任务的全流程智能处理。

更像人写的写法：

> 用户收到重要会议邮件后，OfficeAgent 先做三件事：提取对方问题，检查现有材料能不能回答，标出还缺哪些来源。发送邮件、创建日程、共享文档只生成草稿，用户确认后再执行。

AI 感偏重：

> 出行管家基于多维场景理解，联动差旅准备、交通接驳、机场通行和离线内容 Agent，形成完整出行服务闭环。

更像人写的写法：

> 系统识别到用户下周一早班机后，先生成一张出行卡片：航班时间、出门建议、待准备事项和离线资料。闹钟、叫车、下载文件都只给建议和确认页，Demo 阶段不做真实下单。
