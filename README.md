# Agent PRD Writer zh

这是一个用于撰写和整理 Agent 产品 PRD 的 Codex skill。

它适合处理这类需求：一个产品里有一个或多个 Agent，需要读取上下文、生成结果、调用工具，甚至在用户确认后执行动作。常见场景包括办公助手、会议助手、出行管家、销售线索助手、审批助手、跨系统任务处理工具等。

这个 skill 的重点是帮助产品负责人把一件事讲清楚：这个 Agent 要替用户处理什么任务，首版先做哪条主线，哪些动作需要用户确认，工程团队拿到文档后还缺什么才能开工。模板只作为辅助，不要求每个章节都写满。

## 它能做什么

- 把业务背景、demo、截图、现有草稿或研究材料整理成 Agent PRD。
- 判断产品更适合做成单一 Agent，还是拆成 Agent 套件。
- 写清用户看到的入口、Agent 的职责、关键流程、权限确认、异常处理和验收标准。
- 在信息不足时保留 `待确认`，不把未知的接入、数据、指标写成已经确定的方案。
- 当用户要求“开发可开工”时，追加工程契约和开发交付包，包括 schema、工具调用、Agent 交接、状态转移、日志审计和测试计划。
- 帮助文档少一点模板味，多保留真实业务词、真实输入、项目取舍和当前阶段的处理方式。

## 适合什么时候用

可以在这些情况下使用：

- 已经有一个 Agent 想法，需要整理成产品文档。
- 已经有 demo，希望补成团队能继续推进的 PRD。
- 现有 PRD 太像普通功能需求，需要补 Agent 角色、工具、确认和失败恢复。
- 一个产品里塞了太多职责，需要判断是否拆成多个 Agent。
- 工程团队要开工，需要把产品规则补成更明确的接口、状态和测试要求。

不太适合的情况：

- 只是普通页面、按钮、表单或后台配置需求。
- 只需要写营销介绍、概念稿或对外宣传文案。
- 还没有明确业务场景，只想泛泛讨论“做一个智能助手”。

## 输出方式

默认输出 PM 可负责的 Agent PRD，主要讲产品目标、用户场景、Agent 角色、关键流程、权限确认、验收、风险和版本计划。

只有用户明确要求“可以直接开发”“工程可落地”“开发可开工”时，才会展开工程契约和开发交付包。这样做是为了避免主文档被 schema、日志、测试矩阵压得太重。

## 文件结构

- `SKILL.md`：skill 的触发条件、工作方式、写作规则和质量检查。
- `references/product-patterns.md`：Agent PRD 的表达方式和短样例。
- `references/agent-prd-template.md`：主 PRD 模板。
- `references/engineering-contracts.md`：开发可开工时使用的工程契约。
- `references/development-handoff.md`：转工程 owner 时使用的交付包。
- `references/examples.md`：少量短样例片段。

## 写作原则

这个 skill 默认用中文输出。文字要清楚、直接、专业，但不能像模板自动生成。

写文档时优先保留业务现场里的真实词，比如 mock 邮件、推荐问题、客服微信、一期纯文字、二期卡片、用户 query、静态路线、demo 操作、团队依赖。不要把所有内容都改写成抽象的“能力”“链路”“框架”“治理”。

主文档先讲清产品主线和当前取舍。工程细节放到附录。安全、权限、日志、降级规则只写一次，避免在多个章节里重复。

如果资料里没有证据，就标记待确认。不要补不存在的数据、用户反馈、系统能力或接入状态。
